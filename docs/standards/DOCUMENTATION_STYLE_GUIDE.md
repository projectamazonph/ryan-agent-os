# Documentation Style Guide

## Rules

- Write for an operator who needs to act.
- Use direct language and active voice.
- Define terms before using specialized abbreviations.
- Prefer concrete requirements over aspirational phrases.
- Use numbered requirements when they will be tested.
- Keep one authoritative location for each decision.
- Link related documents instead of duplicating large sections.
- Record dates as `YYYY-MM-DD`.
- Use Mermaid for diagrams that should remain version-controlled.
- Do not use placeholders in accepted documentation.

## Status labels

Use one of:

- Draft
- Proposed
- Accepted
- Superseded
- Deprecated

## Change discipline

A material documentation change must include:

- What changed
- Why it changed
- Which requirements or decisions are affected
- Whether implementation work is required
